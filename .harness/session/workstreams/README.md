# Workstreams

이 폴더는 요청 단위 새 Codex 창에서 업무 유형별 읽을거리를 좁히기 위한 소비자 프로젝트용 진입점입니다.

새 대화창은 하나의 풀스택 담당자로 봅니다. 사용자는 첫 메시지에 업무 내용만 적고, 에이전트가 요청을 분류해 필요한 workstream 파일과 프로젝트 문서만 골라 읽습니다. 기획, 디자인, 웹 프론트 래퍼, iOS 네이티브 래퍼, Supabase/Auth/Postgres/Edge Function, OpenAI 코칭, 배포/운영 판단이 한 요청의 완료 조건에 함께 들어 있으면 같은 창이 전체를 관리합니다.

기존 workstream 파일은 창 분리 기준이 아니라 읽을거리 라우팅 인덱스입니다. 창을 여러 역할로 상시 유지하지 않습니다.

일반론 가이드는 [Workstream Chat Splitting Guide](../../documentation/workstream-chat-splitting-guide.md)를 참고하되, PaceLAB에서는 아래 요청 단위 운영 기준을 우선합니다.

## 요청 단위 운영 기준

- 사용자는 새 요청마다 새 창을 열고 업무 내용을 바로 적습니다. workstream id를 직접 붙일 필요는 없습니다.
- 에이전트는 요청을 보고 관련 workstream 파일을 하나 이상 고른 뒤, 그 안의 "먼저 읽을 문서"와 관련 프로젝트 문서만 읽습니다.
- 한 요청 안에서 기획, UI/UX, 러닝 로직, AI 코칭, HealthKit/iOS, Supabase, 부상관리, 하네스 운영이 함께 필요하면 같은 창이 풀스택 담당자로 전체를 관리합니다.
- 모든 요청 시작 시 기본 제품 표면을 먼저 떠올립니다: 웹 프론트 래퍼(`src/**`, GitHub Pages, iOS WebView 화면), iOS 네이티브 래퍼(`/Users/smart-tn-083/practice/RunningCoach`), Supabase 백엔드(`supabase/**`, Auth/Postgres/RLS/Edge Function), OpenAI secret/코칭 경계.
- 요청이 특정 표면 하나만 언급해도 사용자 흐름이 다른 표면과 연결되면 같은 목표 안에서 계약, 저장, 표시, 배포/캐시 영향까지 함께 검토합니다.
- 완료 책임 창은 기본적으로 현재 요청을 받은 새 창입니다. 다른 상시 workstream 창으로 책임을 이관하지 않습니다.
- 정식 개발 작업이면 에이전트가 직접 GitHub Issue를 만들거나 기존 Issue를 확인하고, Issue 전용 worktree와 branch에서 작업합니다.
- 사용자가 동시에 여러 업무를 요청하면 업무마다 Issue, worktree, branch를 분리합니다. 같은 창이 여러 요청을 순차 관리할 수는 있지만, 파일 변경과 커밋 범위는 Issue별 worktree 격리 원칙을 유지합니다.
- 서로 독립적인 여러 업무를 한 창에서 동시에 구현하지 않습니다. 우선순위를 정하거나 Issue를 나눈 뒤 각 worktree에서 하나씩 처리합니다.
- 인수인계는 긴 작업 중단, 새 창으로 재개, 미커밋 변경 보존, 외부 수동 조치 대기처럼 필요한 경우에만 사용합니다.
- GitHub Project는 작업 추적과 상태판으로 사용하지만, 창끼리 자동 협업한다고 가정하지 않습니다.

## 새 라우팅 항목 추가 기준

기존 01~08 읽을거리로 처리하기 어려운 새 도메인이 반복적으로 등장하면, 하네스/운영 기준 변경 요청에서 새 라우팅 파일 추가를 검토합니다.

새 라우팅 항목은 아래 조건 중 하나 이상을 만족할 때 추가합니다.

- 기존 라우팅 항목의 제외 범위에 반복적으로 걸린다.
- 별도 도메인 규칙, 데이터 계약, 검증 기준, 사용자 인터뷰가 필요하다.
- 여러 기존 항목에 걸쳐 있지만 별도 읽을거리 묶음이 있어야 맥락이 안정된다.
- 한두 번의 임시 작업이 아니라 장기적으로 반복될 가능성이 높다.

새 라우팅 항목을 추가할 때는 다음을 갱신합니다.

- workstreams 폴더의 `NN-name.md` 새 파일
- 이 README의 `읽을거리 라우팅 목록`
- `.harness/project/workflow-rules.md`의 작업 유형 목록과 시작 문서 표
- 필요하면 `.harness/session/project-memory.md`, `.harness/session/decision-log.md`, `.harness/session/next-session-reminder.md`

새 라우팅 파일은 기존 파일과 같은 구조를 사용하되, "담당 범위"는 창 분리 강제가 아니라 읽을거리와 검토 범위로 해석합니다.

## 공통 시작 문구

```text
<업무 요청을 바로 작성>
```

에이전트는 첫 요청을 보고 이 README와 필요한 workstream 파일을 스스로 선택합니다. 사용자가 명시적으로 라우팅을 지정하고 싶을 때만 아래처럼 줄여 쓸 수 있습니다.

```text
이 요청은 06-healthkit-ios 중심으로 처리해줘.
```

창 제목만으로 자동 감지한다고 가정하지 않습니다. 현재 Codex 어댑터는 사용자 프롬프트 본문에 포함된 업무 설명과 workstream id만 안정적으로 감지합니다.

이미 열려 있는 창에 최신 운영 기준을 다시 적용해야 할 때는 아래 문구를 붙여넣습니다.

```text

```

Codex hook은 위와 같은 새로고침 문구를 감지하면 이 README와 업무별 라우팅 기준을 다시 읽으라는 안내를 주입합니다.

## 완료 책임 창

모든 업무 요청은 시작할 때 `완료 책임 창`을 하나 정합니다. 완료 책임 창은 업무 목표, 완료 조건, 후속 workstream 인수인계, 최종 리뷰, 검증 후보 정리를 소유합니다.

- 기본 완료 책임 창은 현재 요청을 받은 새 창입니다.
- 여러 업무 유형을 거치는 요청도 현재 창이 전체 완료 책임을 유지합니다.
- workstream 파일은 완료 책임 이관 기준이 아니라 읽을거리 라우팅 기준입니다.
- 사용자가 명시적으로 특정 중단점이나 별도 handoff를 요청하지 않으면, 현재 창이 기획, 디자인, 구현, 네이티브, Supabase, 검증, PR, merge/deploy 확인까지 관리합니다.
- 완료 책임이 불명확한 경우는 요청 자체가 여러 독립 업무를 섞었거나 외부 수동 결정이 필요한 경우입니다. 이때는 먼저 Issue와 worktree 분리 또는 사용자 결정을 정리합니다.

완료 책임 창은 모든 후속 창의 결과를 모은 뒤 남은 리스크, 미해결 질문, 최종 검증 후보, 커밋 후보 범위를 정리합니다. PaceLAB MVP 단계에서는 단순 확인, 검토, 조사, 기획 질문이 아닌 구현/버그/운영 요청을 사용자가 맡긴 경우 중간 확인을 기본 대기점으로 두지 않습니다. 명시적 보류 지시가 없고 완료 책임 창 안에서 해결 가능한 업무라면 검증, commit, push, PR, main 머지, 배포 확인까지 이어서 진행한 뒤 사용자에게 완료 확인을 요청합니다. 단, GitHub Issue/Project를 `Done`, `Closed`, 또는 이에 준하는 최종 완료 상태로 변경하는 것은 사용자의 명시 지시가 있을 때만 수행합니다.

사용자가 완료 처리, 최종 완료, 닫기 등 명시적 완료 지시를 하면 완료 책임 창은 Issue를 닫기 전에 기준 작업트리 `/Users/smart-tn-083/practice/run-ai`의 `main`을 `origin/main` 최신 상태로 맞춥니다. 기준 작업트리에 미커밋 변경이 있으면 임의로 stash/reset하지 않고 먼저 보고합니다.

사용자가 `검토만`, `조사만`, `PR까지만`, `커밋하지 마`, `배포하지 마`, `여기서 멈춰`처럼 중단점을 지정하면 그 지시를 우선합니다. 완료 책임이나 요청 분해가 불명확하면 넓은 작업을 진행하기 전에 Issue/worktree 분리 기준을 확정합니다.

## 업무 피로도

요청 단위 창은 작업 시작, 긴 작업 종료, 인수인계 전에 현재 창의 `업무 피로도`를 자가진단합니다. 정식 GitHub Issue 작업이면 Project 필드 `업무 피로도`를 `fresh`, `normal`, `tired`, `reset-needed` 중 하나로 맞춥니다.

이 값은 정확한 토큰 수나 내부 기억 품질 측정값이 아닙니다. 대화가 길어져 workstream, Issue, branch, 완료 조건을 혼동할 가능성이 커졌는지 판단하는 운영 신호입니다.

- `fresh`: 새 창이거나 맥락이 짧습니다.
- `normal`: 현재 Issue를 계속 처리해도 됩니다.
- `tired`: 현재 Issue 마무리는 가능하지만 새 작업을 섞지 않습니다.
- `reset-needed`: 넓은 새 작업을 시작하지 않고 handoff를 남긴 뒤 새 요청 창에서 재개합니다.

`tired` 또는 `reset-needed`로 바뀌면 Issue 댓글에 이유와 권장을 실제 Markdown 줄바꿈으로 남깁니다. `reset-needed` 상태의 창은 다음 요청을 새로 받지 않고, 새 창에서 이어갈 수 있도록 짧은 시작 문구와 현재 Issue/branch/worktree 상태를 남깁니다.

## 여러 업무 유형이 섞인 요청

여러 업무 유형이 필요한 요청이라도 기본적으로 같은 창에서 처리합니다. 이 창은 기획자, 디자이너, 웹 개발자, 네이티브 개발자, Supabase 담당자 역할을 함께 수행합니다.

- 읽을거리는 관련 workstream 파일 여러 개를 선택해 좁힙니다.
- 웹 프론트 래퍼, iOS 네이티브 래퍼, Supabase/Edge Function 사이의 데이터 계약은 요청 창이 직접 확인합니다.
- 구현은 하나의 정식 Issue가 하나의 제품/버그 목표를 나타내는 한 같은 Issue worktree에서 처리합니다.
- 독립적인 목표가 여러 개면 parent/child보다 먼저 별도 Issue와 별도 worktree로 나눕니다.
- 한 요청의 완료 조건 안에서 UI, 로직, DB, Edge Function, iOS 변경이 함께 필요하면 같은 Issue/PR에 묶을 수 있습니다.
- 단, 동시에 여러 사용자 요청을 처리하거나 서로 독립적으로 배포 가능한 목표가 섞이면 Issue/worktree/branch를 분리합니다.

## 범위 분리와 인수인계

범위 밖 처리는 "다른 workstream 창으로 넘김"이 아니라 "현재 요청의 완료 조건에 포함되는가"와 "별도 Issue/worktree로 분리해야 하는가"로 판단합니다.

- 현재 요청의 완료 조건에 필요한 작업이면 같은 창에서 계속 처리합니다.
- 독립적인 새 목표이거나 동시에 진행해야 하는 별도 업무라면 새 Issue와 새 worktree/branch를 만든 뒤 섞지 않습니다.
- 외부 수동 확인, 긴 중단, 컨텍스트 리셋이 필요하면 인수인계를 남깁니다.
- 인수인계 문구에는 다음 창이 `git status --short`, `git diff`, 필요 시 `git diff --staged`로 현재 작업트리 변경분을 먼저 확인하라는 문장을 포함합니다.

후속 창에 붙여넣는 인수인계 문구는 기본적으로 일회성 임시 전달물입니다. 진행 중에는 문서화를 늘리지 않고, 최종 완료 승인 시점에 남길 내용만 정리해 문서화합니다. 다만 아래 중 하나에 해당하면 다음 창이 이어받을 수 있도록 진행 중에도 문서에 최소 내역을 남깁니다.

- 여러 창이 이어서 알아야 하는 최신 상태나 pending 작업: `.harness/session/active-context.md` 또는 `.harness/session/next-session-reminder.md`
- 이후에도 유지될 결정, 예외, 충돌 해결 이유: `.harness/session/decision-log.md`
- 반복되는 도메인/아키텍처/워크플로우 기준: `.harness/project/*`
- 사용자 확인 없이는 확정할 수 없는 질문: `.harness/session/developer-input-queue.md`

최종 완료 승인 시에는 임시 인수인계 문구 중 실제로 남길 가치가 있는 내용만 위 문서들로 정리하고, 단순 중간 전달 문구는 남기지 않습니다.

완료 전 창 이동은 커밋 없이 진행할 수 있습니다. 현재 요청에서 별도 Issue/worktree로 분리해야 하면 아래 형식으로 정리합니다.

```text
이 작업은 현재 요청과 분리된 별도 Issue/worktree가 필요합니다.
새 요청 창에 아래 내용을 붙여넣어 이어가세요.

작업 유형: <업무 라우팅 후보>
이전 창: <현재 창 이름>
완료 책임 창: 새 요청 창
작업 상태: 완료/커밋 전 진행 중
먼저 확인할 것:
- `git status --short`
- `git diff`
- 필요 시 `git diff --staged`
현재까지 확인한 내용:
- ...
넘기는 이유:
- ...
관련 파일:
- ...
아직 남은 판단/구현:
- ...
완료 조건:
- ...
주의:
- 현재 요청과 변경을 섞지 말고 별도 Issue/worktree/branch에서 처리한다.
```
