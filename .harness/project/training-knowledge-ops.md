# Training Knowledge Operations

PaceLAB의 훈련 지식 보관소는 AI 코칭 품질을 높이기 위한 운영 데이터다. 사용자가 앱에서 지식화 검토 요청을 저장해도 OpenAI API를 자동 호출하지 않는다. 요청은 검토 대기열에 쌓이고, 개발/운영자가 명시적으로 처리할 때만 조사와 구조화를 진행한다.

## 기본 원칙

- 앱의 `지식화 검토 요청` 저장은 Supabase `training_knowledge_requests` insert만 수행한다.
- 요청 저장 시 OpenAI, 웹 검색, 요약, 임베딩 생성, Edge Function 호출을 자동 실행하지 않는다.
- 승인 전 요청은 AI 코칭 컨텍스트에 들어가지 않는다.
- 코칭에는 `training_methods`, `training_prescription_rules`, 승인된 `training_knowledge_chunks`만 사용한다.
- 원문 전문, 책/유료 콘텐츠의 긴 발췌, 저작권이 있는 문서 본문은 저장하지 않는다.
- 저장 단위는 출처 메타데이터, 짧은 요약, 적용 조건, 처방 규칙, 금기/주의 조건이다.

## 요청이 적을 때의 처리

개발 중 요청이 적으면 Codex가 수동 검토한다.

1. Supabase `training_knowledge_requests`에서 `requested` 상태를 조회한다.
2. 요청 제목, 출처 URL, 사용자가 적은 메모를 확인한다.
3. 공식 문서, 논문, 저자/코치의 1차 자료를 우선 조사한다.
4. 훈련법이 현재 활성 목표와 관련 있는지 판단한다.
5. 원문 저장 없이 PaceLAB 처방에 필요한 구조화 규칙으로 변환한다.
6. `training_knowledge_sources`, `training_methods`, `training_prescription_rules` migration 또는 seed를 작성한다.
7. 요청 상태를 `approved`, `rejected`, 또는 `reviewing`으로 갱신한다.
8. `coach-run` 컨텍스트가 새 규칙을 목표 거리와 세션 타입에 맞게 검색하는지 확인한다.

## 요청이 많아질 때의 운영 확장

요청이 반복적으로 쌓이면 앱 기능보다 먼저 운영 도구를 분리한다.

1. 관리자 전용 목록
   - `requested`, `reviewing`, `approved`, `rejected` 상태별 필터
   - 중복 요청 병합
   - 출처 URL 유효성 확인
   - 승인/반려 사유 기록

2. 배치 검토 CLI
   - `training_knowledge_requests`를 상태별로 조회
   - 중복 제목/URL 그룹핑
   - 검토 후보를 Markdown 리포트로 생성
   - 승인된 항목만 migration 초안으로 변환

3. 반자동 지식화
   - 사용자가 명시적으로 승인한 검토 작업에서만 OpenAI를 호출한다.
   - 호출 단위는 요청 1건 또는 배치 1묶음으로 제한한다.
   - 모델 입력에는 필요한 출처 요약과 사용자 요청만 넣고, 긴 원문 전문을 반복 주입하지 않는다.
   - 결과는 바로 승인하지 않고 사람이 확인한 뒤 `approved`로 바꾼다.

4. RAG/벡터 검색
   - 지식이 수십 개 이상으로 늘어나고 구조화 rule만으로 검색 품질이 떨어질 때 도입한다.
   - `training_knowledge_chunks`에는 승인된 짧은 요약 chunk만 저장한다.
   - embedding은 chunk 승인 시 1회 생성하고, 코칭 요청마다 새로 만들지 않는다.
   - 코칭 컨텍스트에는 검색된 chunk 중 소수만 넣는다.

## 벡터DB 전환 기준

초기에는 구조화 rule 검색이 우선이다. 벡터 검색은 아래 조건 중 여러 개가 동시에 나타날 때 도입한다.

- 승인된 훈련법/문헌이 20개 이상이고 세션 타입/목표 거리만으로 관련 규칙을 좁히기 어렵다.
- 같은 훈련법 안에 목표 거리, 훈련 단계, 부상 상태, 더위 대응처럼 조건이 많이 갈라진다.
- 사용자가 자연어로 “마프랑 80/20 중 지금 내 상황에 뭐가 맞아?”처럼 방법론 비교 질문을 자주 한다.
- 코칭에서 필요한 근거가 `TrainingPrescriptionRule` 하나로 표현되지 않고 여러 요약 chunk를 조합해야 한다.
- 승인된 지식은 늘었는데 AI 컨텍스트에 넣을 수 있는 토큰 예산 때문에 전체 요약을 넣기 어렵다.
- 구조화 rule 검색 결과가 너무 많아져 코칭 품질이 흔들리거나 비용이 증가한다.

도입하지 말아야 하는 경우도 명확히 둔다.

- 승인 지식이 적고, 현재처럼 MAF/VDOT/80:20/Strides/Tempo 정도의 규칙 검색으로 충분하다.
- 원문 전문을 저장해야만 검색이 가능하다고 판단되는 경우.
- 사용자가 비용 증가를 원하지 않는 경우.
- embedding 생성/재생성 운영 주체가 정해지지 않은 경우.

## 벡터DB 전환 방법

Supabase Postgres의 `pgvector`를 기본값으로 사용한다. 별도 벡터DB 서비스를 붙이기 전에 Supabase 안에서 해결한다.

1. 승인 지식만 chunk화
   - 대상은 `approved = true`인 source/method/rule이다.
   - chunk는 원문이 아니라 PaceLAB이 만든 짧은 요약과 적용 조건이다.
   - chunk 하나는 하나의 판단 단위만 담는다. 예: “MAF Easy 하향 조건”, “10K Tempo 심박 상한”, “더위에서 Easy 판정”.

2. embedding 생성
   - 승인 또는 수정 시점에 1회 생성한다.
   - 코칭 요청마다 새 embedding을 만들지 않는다.
   - embedding 모델과 차원은 migration과 운영 문서에 고정한다.
   - embedding 실패 시 chunk는 검색 제외 상태로 두고 코칭은 구조화 rule fallback을 사용한다.

3. 검색 쿼리 구성
   - activeGoal 거리, selectedRun type, 사용자 메모, activeInjury, 날씨/더위 신호를 짧은 검색 문장으로 만든다.
   - 먼저 구조화 필터를 적용한다. 예: `goal_distance in ('all','10K')`, `session_type in ('all','Tempo')`.
   - 그 뒤 벡터 유사도 상위 chunk만 고른다.

4. 컨텍스트 주입
   - 상위 3~5개 chunk만 AI 코칭 컨텍스트에 넣는다.
   - source title, method name, 적용 조건, 처방 요약만 넣는다.
   - 저작권 보호를 위해 긴 인용과 원문 발췌를 넣지 않는다.

5. 결과 검증
   - 같은 세션에 대해 구조화 rule만 썼을 때와 벡터 검색을 추가했을 때의 코칭 차이를 비교한다.
   - 검색된 chunk가 목표/세션과 무관하면 chunk taxonomy 또는 필터를 수정한다.
   - 비용, 응답 시간, 코칭 품질 중 하나라도 나빠지면 구조화 rule fallback을 기본으로 되돌린다.

## 벡터 검색 런타임 규칙

- 벡터 검색은 코칭 근거 보강용이다. 처방의 1차 근거는 여전히 구조화 `TrainingPrescriptionRule`이다.
- 벡터 검색 결과가 없으면 코칭을 실패시키지 않는다.
- 선택 세션의 데이터, activeGoal, activeInjury, 최근 RunLog가 승인 지식보다 우선한다.
- 검색된 chunk는 “참고한 코칭 기준”으로만 쓰고, 사용자의 실제 상태와 충돌하면 사용자 데이터가 우선한다.
- vector index 재생성, embedding 모델 변경, chunk schema 변경은 migration과 함께 처리한다.

## 승인 기준

- 현재 목표 거리와 직접 관련이 있는가.
- 초보/중급 개인 러너에게 적용 가능한가.
- 부상, 더위, 회복 같은 제한 조건을 함께 설명할 수 있는가.
- 세션 타입별 처방으로 바꿀 수 있는가.
- 상향/유지/하향 조정 조건을 만들 수 있는가.
- 출처가 공식 자료, 논문, 저자/코치 1차 자료, 또는 신뢰할 만한 해설 자료인가.

## 반려 기준

- 출처가 불명확하거나 광고성 콘텐츠 위주다.
- 원문을 그대로 저장해야만 의미가 유지된다.
- 엘리트 고볼륨 전제라 현재 사용자 목표에 직접 적용하기 위험하다.
- 의료 진단이나 치료 영역에 가깝다.
- 기존 승인 지식과 중복되며 새 처방 규칙을 만들 수 없다.

## 상태 의미

- `requested`: 사용자가 앱에서 저장한 검토 대기 요청이다.
- `reviewing`: Codex 또는 운영자가 조사 중인 요청이다.
- `approved`: 구조화 지식으로 반영되어 코칭에 활용 가능한 요청이다.
- `rejected`: 현재 기준으로 지식화하지 않기로 한 요청이다.

## 비용 통제

- 사용자의 요청 저장은 무료 동작이어야 한다.
- 코칭 요청마다 전체 지식 문서를 다시 넣지 않는다.
- 조사/요약/임베딩은 승인 워크플로우에서만 발생시키고, 코칭 런타임 비용과 분리한다.
- 코칭 런타임은 activeGoal, selectedRun, sessionType에 맞는 규칙만 가져온다.
- 긴 훈련법 설명은 지식 보관소 화면에 보여주더라도 AI 컨텍스트에는 짧은 처방 규칙만 넣는다.

## 자율 지식 소싱 가드레일 (에이전트 자율 웹소싱 금지) — 2026-06-15 결정

"자기진화"를 두 축으로 분리한다. 위험한 자동 경로만 차단하고, 안전한 자율 상향은 이미 구현된 (A)가 담당한다.

- **(A) 적응(adaptation) — 허용·구현 완료, 결정론적.** 외부 지식 없이 **사용자 run 데이터만으로** 처방 경계를 스스로 상향한다. Tempo/Easy/Long Run drift/Recovery 적응값(추정→검증→채택, 상향만·안전망 cap), ProgressionCriteria 평가, Phase 자동전환(사용자 확인 필요). 사용자 레벨/코칭 분석 기반 "자율 상향"은 이 결정론 엔진으로 한다. 외부 콘텐츠를 끌어오지 않는다.
- **(B) 지식확장(knowledge expansion) — 사람 승인 게이트 필수.** 라이브러리에 새 훈련법 *종류*를 추가할 때만 외부 출처가 필요하다. 에이전트는 **제안자(draft)** 역할까지만 한다.

금지(확정):

- 에이전트가 트리거(레벨업·분석 결과 등)에 반응해 **자율로 웹을 서칭 → 원문 저장 → 승인(`approved`) 없이 코칭에 자동 반영**하는 풀-자동 경로는 금지한다.
- coach-run 컨텍스트에는 **승인된 구조화 규칙만** 주입한다. 자유 형식 웹 텍스트를 프롬프트에 직접 넣지 않는다(간접 프롬프트 인젝션·데이터 포이즈닝 차단).
- 원문 전문/유료 콘텐츠 본문 저장 금지(기존 원칙 유지). 저장 단위는 출처 메타 + 짧은 요약 + 처방 규칙 + 금기 조건.

허용되는 안전 경로(반자동):

1. 트리거 감지 시 에이전트는 `training_knowledge_requests`에 **후보 draft만** insert한다(비용 0, 출처 메타 + 요청 사유). 자동 OpenAI/웹검색 호출 없음.
2. 조사·구조화는 명시 승인된 검토 작업에서만 수행하고, 결과는 `approved=false` staging으로 둔다(`reliability`/`license_note` 필수 기입).
3. 사람(또는 명시 승인된 배치)이 `approved=true`로 승격해야 coach-run에 반영된다.

근거 위험: ①부상/안전(검증 안 된 처방 자동 적용) ②간접 프롬프트 인젝션/데이터 포이즈닝 ③저작권 ④환각/출처 신뢰 ⑤감사·롤백 불가. 상세 결정 이력은 `.harness/session/decision-log.md` 2026-06-15 항목.

## 검증 체크리스트

- 요청 저장 시 브라우저 네트워크에서 OpenAI 호출이 없어야 한다.
- 요청 저장 실패와 승인 지식 조회 실패가 UI에서 구분되어야 한다.
- 승인 전 요청은 `coach-run` context.trainingKnowledge에 포함되지 않아야 한다.
- 새 지식 추가 후 `npm run build`, `npm run test:run`, 필요 시 `npm run e2e`, `npm run harness:check`를 통과해야 한다.
- Supabase migration은 RLS 정책과 함께 작성해야 한다.
