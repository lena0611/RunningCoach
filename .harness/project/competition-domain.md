# 엔터테인먼트 경쟁 도메인 기획

PaceLAB의 엔터테인먼트 도메인 아래 `경쟁` 하위 도메인의 제품 기준 문서입니다. 러너가 기록 경쟁을 **이어폰 음성**으로 경험하게 하는 기능군을 정의합니다.

- 관련 Issue: [#67 엔터테인먼트 경쟁 도메인과 레이스 기능 기획](https://github.com/lena0611/RunningCoach/issues/67) (type:research)
- 1차 기능: `나와의 대결` (본 문서에서 스펙까지 정의)
- 2차 기능: `실시간 레이스` (개요와 선택 기준만, 후속 Research/Decision으로 분리)
- 데이터 원칙: 실시간 위경도(GPS 좌표) **송수신 미사용**. 누적거리·시간·순위·참가자 식별자 등 최소 데이터만 사용.

이 문서는 장기 제품 기준입니다. 진행 상태와 할 일은 GitHub Issue/Project에 둡니다.

---

## 1. 도메인 정의

```
엔터테인먼트
└─ 경쟁 (competition)
   ├─ 나와의 대결   (self-race / 가상 파트너 · 고스트)  ← 1차, 온디바이스, 서버 불필요
   └─ 실시간 레이스 (live-race / 라이브 대결방)         ← 2차, 라이브 서버 필요
```

- 두 기능의 공통 경험: **달리는 중에 화면을 보지 않고도, 이어폰 음성으로 현재 경쟁 상황(앞섰는지/뒤졌는지, 차이)을 듣는다.**
- 두 기능의 결정적 차이: 비교 대상이 `과거의 나`(로컬 데이터)인지 `다른 사용자`(라이브 서버)인지. 이 차이가 서버·실시간·인증·프라이버시 복잡도를 가른다.
- 그래서 1차는 서버 없이 온디바이스로 끝낼 수 있고, 2차만 라이브 서버 결정이 필요하다.

## 2. 1차 기능: `나와의 대결` (Self-Race)

### 2.1 사용자 가치와 핵심 시나리오

- 혼자 달려도 "오늘의 나 vs 과거의 나"라는 경쟁 긴장감과 페이스 가이드를 동시에 얻는다.
- 화면을 계속 볼 수 없는 러닝 특성상, 핵심 피드백은 **음성**이다. 화면은 보조.
- 핵심 시나리오:
  1. 러너가 비교할 **과거 기록 1건**(또는 기준 페이스)을 고른다.
  2. 달리기를 시작하면, 앱이 같은 코스를 달렸던 `과거의 나`(고스트)와의 차이를 실시간 계산한다.
  3. 설정한 주기/이벤트마다 이어폰으로 음성 안내한다. 예: "과거의 나보다 12초 앞서고 있습니다. 1km 페이스 5분 30초."
  4. 종료 시 최종 결과(이겼는지, 누적 시간차, 구간별 우열)를 요약한다.

### 2.2 비교 모델 — 고스트(가상 파트너)

핵심 패턴은 Garmin Virtual Partner / 고스트 러너와 같다. **누적거리 기준 시간차**가 1차 비교 단위다.

- 과거 기록에서 `누적거리 → 경과시간` 곡선 `T_past(d)`를 만든다.
- 현재 달리기의 라이브 `누적거리 d_now`와 `경과시간 t_now`를 받는다.
- 같은 누적거리에서의 차이를 계산한다:
  - **시간차** `Δt(d_now) = t_now − T_past(d_now)` → 음수면 앞섬, 양수면 뒤짐.
  - **거리차** (같은 경과시간 기준) `Δd(t_now) = d_now − D_past(t_now)` → 양수면 고스트보다 앞.
  - **현재 페이스차** = 현재 순간 페이스 − 같은 지점의 과거 페이스.
- 기본 안내는 **시간차**로 한다("12초 앞/뒤"). 가장 직관적이고, 음성 한 문장에 담기 쉽다.

비교 단위 옵션(설정으로 노출):
- **고스트(연속) 모드 (기본/MVP)**: 과거 페이스 곡선을 그대로 고스트로 사용. 과거의 내가 빨라졌다 느려졌던 그대로 경쟁.
- **균등 페이스 모드**: 과거 기록의 평균 페이스를 일정하게 적용한 고스트(또는 사용자가 목표 페이스 직접 입력). 페이스 곡선이 부실한 기록에 대한 fallback.
- **랩 단위 비교 (보조 표시)**: 1km 등 랩 경계에서 "이번 1km, 과거보다 8초 빨랐습니다" 누적이 아닌 구간 우열.
- **라우트 단위 비교 (후속)**: 같은 코스를 GPS 형태 매칭으로 정렬해 지점별 비교. 정밀하지만 매칭 난이도가 높아 MVP 비목표.

### 2.3 입력 데이터

과거 기록은 이미 저장된 `RunLog`를 그대로 쓴다 (`src/entities/run/model.ts`).

- 고스트 곡선 `T_past(d)` 생성 우선순위:
  1. `metricSamples[]` (`offsetSec`, `paceSec`) — 페이스 곡선을 적분해 시간↔누적거리 곡선 생성. **가장 현실적인 고스트.**
  2. 부족하면 `laps[]` (`distanceKm`, `paceSec`)로 구간 평균 곡선.
  3. 둘 다 부실하면 `distanceKm` + `durationSec` 기반 **균등 페이스** fallback.
- 과거 기록 선택 기준(UI): 같은/유사 거리, 최근, 개인 최고(PB), 사용자가 직접 선택. MVP는 "목록에서 직접 선택" + "비슷한 거리 추천" 정도.
- `routePoints[]`는 1차에서 **비교에 사용하지 않는다**(라우트 매칭은 후속). 위경도 미사용 원칙과도 일치.

### 2.4 핵심 선행 갭 — 라이브 인-런 추적 (중요)

> **PaceLAB는 현재 달리는 중의 라이브 데이터를 받지 않는다.** HealthKit에서 운동이 **끝난 뒤** `RunLog` 후보를 사후 import하는 구조다(`.harness/project/healthkit-data-contract.md`). `나와의 대결`은 **달리는 중의 누적거리·경과시간 라이브 피드**가 반드시 필요하므로, 이 기능의 최대 선행 작업은 "라이브 인-런 세션" 신설이다.

라이브 누적거리 피드 후보 (iOS 네이티브 `/Users/smart-tn-083/practice/RunningCoach`):
- **CoreLocation (`CLLocationManager`) 백그라운드 위치 → 누적거리 적분**: 야외 GPS 기반. 좌표는 거리 계산에만 쓰고 외부로 보내지 않는다(위경도 미송신 원칙 유지).
- **CMPedometer (걸음 기반 거리)**: GPS 약한 환경 보조, 배터리 유리, 정확도 낮음.
- **HKWorkoutSession 라이브**: watchOS 중심. iOS 단독 라이브 워크아웃 가능 범위는 OS 버전별로 검증 필요(아래 검증 후보).
- 세션 종료 후 정식 기록은 기존 HealthKit import 경로로 `RunLog` 저장(이중 트래킹 방지: 라이브는 경쟁 계산용, 정본 저장은 HealthKit 유지).

이 선행 갭 때문에 `나와의 대결`은 "음성/비교 로직"보다 "**iOS 라이브 트래킹 + 브리지 계약**"이 먼저 풀려야 하는 기능이다. 후속 child 분해에서 이를 1순위 선행으로 둔다.

### 2.5 음성 안내 정책

- **재생 경로**: iOS `AVSpeechSynthesizer`(온디바이스 TTS, 오프라인 가능, 한국어 지원) 우선. 음악/팟캐스트와 공존하도록 `AVAudioSession`을 `.duckOthers`로 설정(안내 시 다른 소리를 잠깐 줄임). 안내 트리거는 웹에서 내려보내고 실제 음성 합성·오디오 세션은 네이티브가 담당하는 브리지 계약이 깔끔하다.
- **안내 주기(선택형)**: 혼자하기는 **러너가 직접 선택**, 여러명이서 하기는 **방장이 선택**(후속). 하나의 주기 모드를 고른다.
  - 거리 기반: **100m / 500m / 1km** 등 (스텝 선택).
  - 시간 기반: **1분 / N분** 등 (스텝 선택).
  - 조용히: 주기 안내 없이 이벤트만.
- **이벤트 기반(주기와 별개, 항상/토글)**: 우열 **역전 시**(앞↔뒤), PB 갱신 임박, 마지막 구간. 주기를 '조용히'로 둬도 역전 안내는 켤 수 있다.
- **문구 템플릿(한국어, 짧게)**:
  - 우세: `과거의 나보다 {n}초 앞서고 있습니다.`
  - 열세: `과거의 나보다 {n}초 뒤처졌습니다. 현재 페이스 {pace}.`
  - 역전: `방금 과거의 나를 따라잡았습니다!` / `과거의 나에게 추월당했습니다.`
  - 랩: `{k}킬로미터 통과, 과거보다 {n}초 {빠름/느림}.`
  - 종료: `완주. 과거의 나보다 총 {n}초 {빠릅니다/느립니다}.`
- 숫자·페이스 표기는 앱 공통 포맷(분:초)을 따른다. 문구가 길어지면 음성이 다음 안내와 겹치므로 한 문장 원칙.

### 2.6 구현 표면

| 표면 | 역할 |
| --- | --- |
| iOS 네이티브 (`RunningCoach`) | 라이브 위치/거리 트래킹, 백그라운드 실행, `AVSpeechSynthesizer` 음성·오디오 세션, 라이브 누적거리·경과시간을 WebView로 브리지 |
| 웹 프론트 (`src/**`) | 과거 기록 선택 UI, 경쟁 설정(주기/모드), 라이브 비교 계산(고스트 곡선·gap·역전 판정), 안내 트리거 생성, 화면 표시(보조), 종료 요약 |
| 데이터 | 과거 비교는 기존 `RunLog` 재사용. 경쟁 결과를 저장한다면 경량 `competition_result` 후보(거리/총 시간차/우열/대상 기록 id) — MVP는 저장 없이 종료 화면 표시만으로도 가능 |
| Supabase Edge | **불필요.** 온디바이스 비교라 secret/서버 호출 없음 |

비교 로직(고스트 곡선, gap, 역전, 포맷)은 순수 함수로 웹 `shared/lib`에 두고 Vitest 단위 테스트 대상으로 한다(계산/포맷 로직 회귀 기준과 일치).

### 2.7 오프라인 / 실패 / 중단 / 재개

- **오프라인**: 과거 기록이 로컬에 있고 비교가 온디바이스이므로 **완전 오프라인 가능**. 네트워크 의존 없음(GPS는 네트워크와 무관).
- **GPS 끊김/튐**: 누적거리 점프 방지(속도 상한 필터), 신호 약하면 안내에 "신호 약함" 상태 반영 또는 직전 페이스 유지. 좌표가 아니라 거리 증분만 신뢰.
- **일시정지/재개**: 사용자가 멈추거나 자동 일시정지 시 경과시간·고스트도 함께 멈춤(고스트는 "달린 거리" 기준이라 정지 중 벌어지지 않음).
- **백그라운드/화면 잠금**: 백그라운드 위치 + 오디오 세션 유지가 핵심. iOS 백그라운드 실행/배터리 제약 검증 필요(아래 검증 후보).
- **앱 강제 종료/크래시**: 진행 상태(현재 누적거리·경과시간·대상 기록)를 주기적으로 영속화해 재개 시 복원 시도. 복원 불가면 "경쟁 중단됨" 안내 후 일반 기록으로 마무리.

### 2.8 1차 MVP 컷 / 비목표

MVP 포함:
- 과거 기록 1건 선택(직접 선택 + 유사 거리 추천).
- 고스트(연속) 모드 + 균등 페이스 fallback.
- 누적거리 기준 시간차 음성 안내(거리 주기 + 역전 이벤트).
- 라이브 누적거리/경과시간 트래킹(선행 갭) + 종료 요약.

MVP 비목표(후속):
- 라우트(GPS 형태) 매칭 비교.
- 여러 고스트 동시 경쟁, 친구 기록과의 비대면 경쟁.
- 경쟁 결과 히스토리/랭킹 저장 화면.
- Apple Watch 전용 앱.

## 3. 2차 기능: `실시간 레이스` (개요 · 후속 분리)

이번 세션 범위 밖. 결론과 선택 기준만 남기고 별도 Research/Decision Issue로 분해한다.

- 흐름: 방 생성/입장 → 방장 시작 → 참여자 간 **누적거리** 기반 순위 → 음성 주기 설정대로 순위 안내 → 종료/결과.
- 송수신 데이터: 라우트 변경 이벤트 또는 5초 주기로 **참가자별 누적거리·경과시간·식별자**만. **위경도 좌표는 송수신하지 않는다.**
- 난이도 핵심: 라이브 서버, 인증/방 권한, 지연/이탈/재접속, 부정행위(거리 조작) 방지, 비용.
- `나와의 대결`에서 만든 라이브 트래킹·고스트 비교·음성 엔진을 그대로 재사용하고, "비교 대상"만 로컬 과거 기록 → 원격 참가자 상태로 바꾸는 구조가 이상적. → **1차를 먼저 하면 2차의 클라이언트 절반이 끝난다.**

## 4. 전용 Watch앱 필요성 (가벼운 결론)

| 조건 | iPhone + 이어폰으로 충분 | Apple Watch 전용 앱 필요/유리 |
| --- | --- | --- |
| 누적거리·경과시간 라이브 | 가능 (CoreLocation/CMPedometer) | 더 정확/저전력, 단독 운동 |
| 이어폰 음성 안내 | 가능 (AVSpeechSynthesizer + AVAudioSession) | 워치 단독 시 워치→이어폰 경로 필요 |
| 화면 글랜스 | 제한적(주머니/암밴드) | 손목에서 즉시 확인 |
| 심박 라이브 | iPhone 단독은 약함 | 워치 HR 센서로 강함 |
| 배터리/백그라운드 안정성 | iPhone 백그라운드 제약 검증 필요 | 워치 운동 세션이 안정적 |

- **결론**: `나와의 대결` 1차 MVP는 **iPhone + 이어폰만으로 가능**하다는 가설로 진행한다. Watch 전용 앱은 (a) 라이브 심박 경쟁, (b) 손목 글랜스, (c) iPhone 없이 단독 운동이 핵심 가치가 될 때 별도 검토. 단, iPhone 백그라운드 위치+오디오의 장시간 안정성·배터리는 **착수 전 검증 항목**(아래)이며, 여기서 막히면 Watch 우선순위가 올라간다.

### 4.1 장기 방향 — 워치가 워크아웃 기록의 본거지 (방향 확정, MVP 밖 / 상세는 Research 이슈)

현재 측정은 Apple 워크아웃/피트니스가 기록 → 우리는 사후 import(읽기만). 장기 지향은 **우리 워치앱이 `HKWorkoutSession`으로 워크아웃을 직접 기록**해 측정 주체가 되는 것이다.

- **정본 구조 유지**: 우리 워치앱이 기록해도 운동은 HealthKit에 저장(sourceName=PaceLAB)되므로 기존 `RunLog` import 계약(§9.1, healthkit-data-contract)이 그대로 유지된다. 우리가 "기록자"까지 겸할 뿐.
- **이점**: 라이브 심박(iPhone 단독 거의 불가)·거리·페이스를 워치 센서로 깨끗하게 확보 → #229의 iPhone CoreLocation 라이브 추적 한계를 더 안정적으로 해소. 배터리/백그라운드도 워치 운동 세션이 유리.
- **비용**: 워치앱은 네이티브 SwiftUI라 WKWebView/Vue 두뇌 재사용 불가. 라이브 루프·고스트 비교를 워치에서 다시 네이티브로 구현(#230 알고리즘 사양은 공유). `xcode-select` 전환 + watch 타깃 추가 + 실기기 서명 필요.
- **❓ 핵심 범위 결정(미정, Research에서 확정)**: 워치앱이 **(A) 레이싱 전용**(훈련은 계속 Apple 워크아웃→import, 우리 워치앱은 레이싱할 때만) vs **(B) 훈련+레이싱 통합**(일상 운동 기록까지 우리 워치앱이 본거지). 
  - 권장 1차 = **(A) 레이싱 전용**: 범위·리스크 작고, 사용자의 일상 드라이버(Apple 워크아웃)를 강제 교체 안 함. `'self-race'` 태그 = "우리 워치앱이 기록한 레이싱"으로 자연 매핑. (B) 통합은 Apple 워크아웃 수준의 완성도(자동일시정지·구간·신뢰성)를 요구해 바가 높음 → 후속 확장.

## 5. 라이브 서버 후보 비교 (가벼운 결론, 2차용)

`실시간 레이스`에만 필요. 5초 주기 누적거리 송수신, 방 단위 브로드캐스트가 요건.

| 후보 | 장점 | 단점/리스크 | 적합도 |
| --- | --- | --- | --- |
| Supabase Realtime (Broadcast/Presence) | 이미 Supabase Auth/RLS 사용 중, 방=channel, Presence로 이탈 감지, 추가 서버 운영 없음 | 메시지 영속/순위 권위 계산은 별도 설계, 대규모 동시성 비용·한도 확인 필요 | **1순위 후보** |

| 별도 WebSocket 서버 | 완전 제어, 권위 순위·부정 방지 로직 자유 | 인프라 신설·운영·secret·비용, MVP 과투자 | 대규모 확장 시 |

- **선택 기준**: 동시 참가 인원, 방 수, 순위 권위 계산 위치(클라 vs 서버), 부정행위 방지 수준, 운영 비용. 소규모 친구 대결부터면 **Supabase Realtime 우선 검증**.
- 위경도 미송신 원칙상 메시지 페이로드는 `{participantId, cumulativeDistanceM, elapsedSec, seq}` 수준으로 최소화.

## 9. 구현 스펙 (child Issues #228~#233)

기획(§1~8)을 구현 계약 수준으로 확정한다. 실제 브리지 패턴(`RunContextWebView.swift` ↔ `src/features/import-healthkit-run/healthKitBridge.ts`)과 `RunLog` 모델(`src/entities/run/model.ts`)에 맞춘다.

### 9.1 경계 — 포그라운드는 웹, 백그라운드 핵심 루프는 네이티브 (확정)
> **WKWebView는 백그라운드(화면 잠금·앱 백그라운드)에서 JS가 정지·스로틀된다.** 폰을 주머니에 넣고 이어폰만으로 달리는 게 핵심 시나리오이므로, **시간 임계 루프(틱 → gap/역전 비교 → 음성 발화)는 네이티브가 직접 돌린다.** 웹에 둘 수 없다.

- **네이티브 = 진실의 출처 + 백그라운드 실행 엔진**: GPS/거리 적분, 백그라운드 실행, 일시정지, 영속화, **고스트 곡선 보관 + gap/역전 계산 + 발화 트리거 + TTS 재생**. 틱 페이로드에 위경도 미포함.
- **웹 = 선택·포그라운드·요약**: 타겟 선택, 설정, **고스트 곡선 생성(시작 시 네이티브로 1회 주입)**, 포그라운드 보조 화면(틱 수신해 표시), 종료 요약. 포그라운드에선 웹도 같은 비교를 표시할 수 있으나 **백그라운드 권위는 네이티브**.
- 고스트 곡선은 정적이라 시작 시 한 번 넘기면 이후 비교는 단순 보간 → 네이티브 계산이 가볍다.
- **단일 사양, 2-surface 구현**: gap/역전/포맷 알고리즘은 #230 순수 함수로 **정의·테스트**(canonical spec)하고, 백그라운드 실행용으로 네이티브가 **같은 알고리즘을 포팅**한다. 둘이 어긋나지 않게 #230이 기준.
- 정본 저장은 기존 HealthKit import 유지(이중 트래킹 금지). 라이브는 경쟁 계산 전용.

### 9.2 타겟 모델 — 거리별 PB 고스트
- 타겟은 "과거 RunLog 1건"이 아니라 **거리별 PB**(개인 최고). PB 산출·업적 등록은 **#181 업적 도메인이 소유**(#228), `나와의 대결`은 **소비만** 한다.
- PB 정의(확정): **출발선부터 누적거리 D(5km 단위)에 가장 빨리 도달한 기록**. 그 D를 소유한 실제 `RunLog`의 곡선이 고스트 입력.
- **PB 사다리 2개 분리(업적 표시용, 유지)**: 업적 PB는 **훈련간 PB**와 **레이싱간 PB**를 상호 배타로 분리해 전시한다. 분리 키 = `RunLog.tags`의 `'self-race'` 포함 여부(§10 태깅). 이 분리는 **업적 표시 전용**이고, 아래 **솔로 타겟 모수와는 별개**다(두 질문을 섞지 않는다).
- **레이싱 모드(확정)**: `레이싱하기` = **나와의 대결**(혼자, 이번 개발 범위) + **크루와 대결**(= `실시간 레이스` §3, 후속 #67). 둘은 같은 competition 프리미티브의 N=1 / N>1 케이스다(고스트 = 라이브 친구의 정적 버전). 데이터 모델은 멀티 전제로 설계한다 — [[decision-log 2026-06-09]] 참고.
- **나와의 대결 = 실력 측정 / "내 베스트 도전"(확정, 2026-06-09).** 솔로는 "지금 내 실력 어디쯤인가"를 확인하는 자기 벤치마크다. 그래서 과거 기록을 상대로 두는 건 "경쟁"이 아니라 "내 최고 페이스 도전".
- **혼자하기 타겟 선택(확정, 2026-06-09 갱신)**: **① 없음**(고스트 없이 자유 타임트라이얼) 또는 **② 내 베스트**(거리별)뿐이다. 내 베스트 **모수 = 훈련·레이싱 전체 통합 최속 1개**(컨텍스트 무관). ※ 이전 "레이싱 세션만/훈련 제외" 결정은 폐기 — 벤치마크엔 컨텍스트 무관 최고가 자연스럽다.
- **내 베스트 한계 인지**: PB 정의가 "출발선부터 누적거리 D 도달 최속"이라 5km 전용 TT가 아니라 "어느 런이든 첫 D km 최속"이다(예: 템포런 초반 포함). 거슬리면 후속 품질필터(D가 총거리 큰 비중인 런만)로 보정.
- **부트스트랩**: 5km↑ 기록이 있으면 첫 레이싱부터 '내 베스트' 타겟이 잡힌다(레이싱 0건이어도 됨). 베스트가 없으면 '없음'만.
- 데이터 흐름: 타겟='없음'이면 고스트 비교 생략(측정만); 타겟='내 베스트'이면 그 베스트 `RunLog` → `buildGhostCurve` → 라이브 비교. 구현: `src/features/live-run/raceTargets.ts`(`listOpponents` = 없음/내 베스트 통합 최속).

### 9.3 신규 브리지 계약 2종 (기존 패턴 확장)
**`runContextLiveRun`** (#229) — web→native: `startLiveRun{sessionId, mode, ghostCurve?, announceConfig, tickIntervalMs?}` (ghostCurve=정적 `{distanceM,elapsedSec}[]`, 시작 시 1회 주입해 백그라운드 비교 가능; 타겟 '없음'이면 ghostCurve 생략). `announceConfig = {periodic:{kind:'distance'|'time'|'silent', stepM?:100|500|1000, stepSec?:60|...}, reversalAlert:boolean}` — 혼자하기는 러너 선택, 여러명이서 하기는 방장 선택(후속). 네이티브 엔진이 이 설정으로 발화 주기를 판정 / `pauseLiveRun` / `resumeLiveRun` / `stopLiveRun` / `requestRecoverableLiveRun`. native→web(`window.RunContextLiveRun.*`): `receiveTick{seq,elapsedSec,cumulativeDistanceM,instantPaceSec,signalState,source}`(~1Hz, 좌표 없음, 포그라운드 표시용) / `receiveGap{timeGapSec,leadState}`(포그라운드 UI용) / `receiveStateChange{state}` / `receivePermission{status}` / `receiveRecoverable{...}|null` / `receiveError{code,message}`. 거리 점프 방어(속도 상한)·gap/역전 비교·발화 트리거는 **네이티브에서**(백그라운드 동작). 포그라운드에선 틱/gap을 웹으로 올려 화면 갱신.

**`runContextSpeech`** (#231) — web→native: `configureAudioSession{duckOthers:true}` / `speak{text,priority,dedupeKey?}` / `cancelSpeech`. 네이티브 `AVSpeechSynthesizer`(ko-KR) + `AVAudioSession .playback[.duckOthers,.mixWithOthers]`. high=interrupt, normal=큐잉.

## 10. 가상레이싱 도메인 분류 (확정)

**가상레이싱은 "훈련의 한 종류"도 "별도 활동"도 아니다. 훈련 분류와 직교(orthogonal)하는 경쟁 도메인의 "수행 모드 + 결과 주석"이다.**

| 질문 | 소유 도메인 | 처리 |
| --- | --- | --- |
| 어떻게/얼마나 강하게 달렸나 (type·부하·부상·추세·코칭) | **훈련**(RunLog 정본) | `inferRunType`이 데이터로 판정. **'Race'로 강제 금지** |
| 무엇과 겨뤘나·이겼나 (타겟 PB·시간차·승패) | **경쟁**(신규) | RunLog에 링크된 경량 `competition_result` |

- `type:'Race'`는 Riegel 예측(`performanceProjection.ts`)·회복 비용·테이퍼·하드세션 부하(`ruleBasedCoach.ts`/`runStats.ts`/`trendInsights.ts`)에 직접 투입되는 무거운 신호다. 가상레이싱을 'Race'로 두면 이 경로가 오염된다 → 금지.
- 정본 활동 = HealthKit→RunLog (type 불변). `RunLog.tags += 'self-race'` 경량 태그가 **이중 목적**: (a) 레이싱 수행 세션 식별, (b) 업적 PB의 훈련간/레이싱간 사다리 분리 키(#228 §9.2).
- 레이싱 결과 = `competition_result{mode:'self-pb',targetPb:{distanceM,elapsedSec,sourceRunId},racedDistanceM,resultGapSec,outcome,linkedRunId,racedAt}`. 종료 후 import된 RunLog와 시간·거리 근접 매칭(기존 `requestRunningWorkoutByExternalId` 패턴 재사용).
- **이중계산 방지**: `competition_result`는 볼륨·부하·추세 집계에 미포함. 업적·동기부여·코칭 인용 전용. PB 갱신 시 #228이 전체 RunLog 재산출로 자동 반영.
